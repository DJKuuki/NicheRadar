# PolyMarket 小众事件机器人方案

## 1. 目标定义

机器人目标不是在 `BTC 5m`、做市、新闻抢跑这些高竞争市场里拼速度，而是在小众、可验证、定价效率较低的预测市场里稳定找 `maker` 型机会。

首个可执行方向：

- 聚焦 `社交媒体 / 娱乐 / 小众事件` 市场
- 只做 `规则清晰、外部证据可验证` 的二元市场
- 只在 `模型价格` 与 `市场价格` 出现足够偏差时挂单
- 默认优先 `maker`，尽量避免 `taker` 成本吞噬 edge

成功标准：

- 单笔交易都有正期望且覆盖费用
- 能持续输出可审计的信号与下单理由
- 能自动暂停失效品类或异常市场

## 2. 不做什么

以下方向直接排除：

- `BTC 5m`、主流 crypto 高流动性方向
- 鲸鱼跟单、地址画像跟单
- 纯新闻抢跑
- 低价尾部彩票
- 靠极薄价差吃结算窗口
- 需要 HFT 级执行的套利

原因很简单：这些方向要么对手太强，要么成本结构错误，要么信号链太长。

## 3. 目标市场画像

机器人只扫描满足以下条件的市场：

- 二元市场，结算规则可结构化解析
- 市场标题和规则可以映射到明确事件
- 距离到期时间 `1-14` 天
- 有一定成交，但不是全站最热门品类
- YES/NO 最优价差合理，允许挂单成交
- 外部证据可以持续更新

优先市场类型：

- 社交媒体账号是否会在某日前发布某类内容
- 某艺人/品牌/节目是否会在某窗口内官宣
- 某事件是否会在固定日期前落地

排除市场类型：

- 规则歧义大
- 需要人工语义解释才能判断结果
- 受单一突发新闻强驱动
- 市场已接近结算且剩余空间不足以覆盖成本

## 4. 核心逻辑

### 4.1 核心思想

不做“猜方向”，而是做“重新定价”。

系统对每个候选市场输出：

- `p_model`: 模型估计概率
- `p_mid`: 市场中位概率
- `edge = p_model - p_mid`
- `net_edge = edge - total_cost - uncertainty_buffer`

只有 `net_edge > 0` 时才允许下单。

### 4.2 概率建模

第一版不需要复杂机器学习，先用规则模型 + 校准就够了。

输入特征建议分四类：

1. 时间特征
- 距离截止时间
- 历史上同类事件通常在窗口前几天发生
- 是否进入高密度发布时间段

2. 账号/主体特征
- 官方账号过去 `7/14/30` 天发帖频率
- 发帖时间分布
- 活跃度变化
- 是否存在临近事件前的典型预热模式

3. 外部证据特征
- 是否存在预告帖
- 是否出现官网占位、预约页、倒计时页
- 是否有合作方同步预热
- 是否出现可靠媒体报道

4. 市场微结构特征
- 当前中位价
- 价差宽度
- 近 `N` 小时成交变化
- 是否出现异常拉盘/砸盘

第一版模型形式：

- `base_rate(event_type)`
- `evidence_score`
- `timing_score`
- `market_penalty`
- `uncertainty_penalty`

最终：

`p_model = sigmoid(base + evidence + timing - penalties)`

### 4.3 事件类型分桶

不要把所有市场混成一个模型。至少拆成三桶：

- `announcement`
- `content_release`
- `social_activity`

每一桶独立统计命中率、平均 edge、成交率和回撤。

## 5. 数据管线

### 5.1 市场数据

需要的数据：

- 市场列表
- 标题、描述、规则、截止时间
- YES/NO orderbook
- 最近成交
- 市场状态变化

用途：

- 过滤候选市场
- 计算可成交价格
- 判断执行成本
- 做回测与复盘

### 5.2 外部证据数据

建议从低复杂度源开始：

- X/Twitter 官方账号页面与时间线
- YouTube 频道更新
- 官方网站新闻页
- RSS/媒体文章页

原则：

- 先抓结构化、低延迟、低歧义数据
- 不依赖长链路 LLM 实时理解
- 证据尽量可缓存、可回放、可人工审计

### 5.3 存储

最小化方案：

- `markets` 表：市场元数据
- `snapshots` 表：盘口快照
- `signals` 表：每次模型输出
- `orders` 表：挂单/成交/撤单记录
- `evidence` 表：外部证据抓取结果

任意一笔交易都要能追溯：

- 当时看到了什么市场状态
- 当时抓到了什么证据
- 为什么算出这个概率
- 为什么在这个价格挂单

## 6. 执行策略

### 6.1 只做 maker-first

默认流程：

- 根据 `p_model` 计算最大可接受买价
- 在不穿透盘口的前提下挂单
- 等待成交
- 超时、盘口移动或 edge 消失则撤单

只在极少数情况下允许 `taker`：

- 剩余 edge 远大于费用
- 事件临近且确定性突然提升
- 市场流动性允许立即退出

否则一律不扫板。

### 6.2 定价

买 YES 时：

- `max_yes_bid = p_model - fee_buffer - uncertainty_buffer`

买 NO 时：

- `max_no_bid = (1 - p_model) - fee_buffer - uncertainty_buffer`

建议初始参数：

- `fee_buffer = 0.02`
- `uncertainty_buffer = 0.03`
- 总安全边际至少 `5` 个百分点

如果做不到，就不下单。

### 6.3 分层挂单

不要一次性打满仓位。

建议：

- 第一层试探仓 `25%`
- 第二层确认仓 `35%`
- 第三层扩张仓 `40%`

每层之间都要重新计算 edge，不能机械补仓。

## 7. 风控

### 7.1 仓位限制

- 单市场最大风险敞口不超过总资金 `2%`
- 单事件类型最大风险敞口不超过总资金 `8%`
- 单主体相关市场总敞口不超过总资金 `5%`

### 7.2 失效检测

以下情况触发暂停：

- 最近 `20` 笔信号的实际命中率显著低于回测区间
- 挂单成交率下降超过阈值
- 平均价差明显缩窄，说明更强对手进入
- 相同信号的平均持仓收益连续转负

### 7.3 市场级熔断

以下情况直接禁用市场：

- 规则文本变更
- 盘口突然极端跳变
- 外部证据源失联
- 市场成交异常集中在少数瞬间

### 7.4 组合级熔断

- 日内亏损超过总资金 `3%` 暂停当日开仓
- 周度亏损超过总资金 `6%` 进入观察模式
- 连续 `3` 天 net edge 为负，停机复盘

## 8. 最小可运行版本

### 8.1 MVP 范围

第一版只做一类市场：

- `社交媒体账号是否会在某日前发布内容`

这样做的原因：

- 结算清晰
- 数据好抓
- 证据容易结构化
- 可以人工复核

### 8.2 MVP 模块

1. `market_scanner`
- 拉取市场列表
- 过滤出目标市场

2. `market_parser`
- 解析标题、规则、截止时间
- 抽取主体、事件类型、时间窗口

3. `evidence_collector`
- 抓目标账号过去一段时间的行为
- 检查是否存在预热信号

4. `signal_engine`
- 产出 `p_model`、置信度、理由

5. `execution_engine`
- 根据盘口决定是否挂单
- 管理撤单和重挂

6. `risk_engine`
- 检查仓位、相关性、熔断条件

7. `logger`
- 记录所有输入、输出和决策

### 8.3 运行频率

- 市场扫描：每 `5-10` 分钟
- 外部证据刷新：每 `15-30` 分钟
- 持仓与订单监控：每 `30-60` 秒

这不是 HFT 系统，重点是稳定、可解释、低成本。

## 9. 回测与验证

### 9.1 不要直接上实盘

先做三层验证：

1. 历史回放
- 用历史市场和外部证据回放信号

2. 影子盘
- 实时出信号但不下单

3. 小资金实盘
- 只开最小仓位验证成交与滑点

### 9.2 核心评估指标

- `net pnl`
- 每笔平均 `net edge`
- 成交率
- 挂单存活时间
- 最大回撤
- 分事件类型命中率
- 分市场类型收益

重点不是胜率，而是 `单位风险下的净收益`。

## 10. 真实可行的技术栈

建议先用简单栈：

- `Python`
- `SQLite` 或 `PostgreSQL`
- 定时任务 + 长轮询
- 轻量 dashboard

不要第一天就上：

- 多 agent
- 重型 LLM 工作流
- 高频撮合优化
- 多账户并行放大

这些都是二阶段问题。第一阶段只有一个目标：验证一个品类在扣除真实成本后是否有稳定正期望。

## 11. 实施顺序

### 阶段一：研究验证

- 收集近一段时间的候选市场
- 建立市场分类与规则解析
- 为每类市场人工定义证据模板
- 做纸面定价和回放

### 阶段二：影子运行

- 接实时市场数据
- 接外部证据抓取
- 每次输出模型价格和建议挂单价
- 不真实下单，只记录理论成交

### 阶段三：小资金实盘

- 单市场极小仓位
- 只做 maker
- 严格执行熔断

### 阶段四：扩大

- 复制到相似市场桶
- 细化参数
- 再考虑多账号

## 12. 一句话原则

这个机器人不应该试图“比全市场更快”，而应该试图“在小众市场里，比手动交易者更稳定地定价”。
